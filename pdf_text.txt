Web Real-Time Communication (WebRTC) is a powerful, open-source framework that
enables peer-to-peer (P2P) audio, video, and data sharing directly within web
browsers, eliminating the need for plugins or native applications. Its core strength lies in
establishing direct connections between clients, which significantly reduces latency
and server costs for streaming media. However, WebRTC cannot discover other peers
or negotiate connection details on its own. This process, known as 'signaling,' requires a
separate server to coordinate communication.
This project combines the frontend power of Next.js and the type safety of TypeScript
with the real-time capabilities of WebSockets to build a robust signaling mechanism.
You will implement the 'offer/answer' model, where one peer sends a session
description (the 'offer'), and another responds with its own (the 'answer'). To traverse
network address translators (NATs) and firewalls, which often obscure a device's true IP
address, WebRTC uses the Interactive Connectivity Establishment (ICE) framework. ICE
employs STUN (Session Traversal Utilities for NAT) servers to discover a peer's public IP
address and TURN (Traversal Using Relays around NAT) servers to relay traffic if a direct
P2P connection fails. For this task, you will configure your application to use publicly
available STUN servers.
The architecture will involve a custom Next.js server that runs a WebSocket instance
alongside the standard Next.js application handler. This allows the same server process
to handle both API/page rendering and real-time signaling events. For multi-peer calls,
you will implement a 'mesh' topology, where every participant establishes a direct P2P
connection with every other participant. While simple to implement for small groups,
this model's connection count grows quadratically, making it important to understand
its scalability limitations.

Implementation Guidelines
These are suggestions to guide your implementation, not strict requirements.
Architecture and State Management
1. Separation of Concerns: Consider creating separate modules or hooks for
managing WebSocket signaling (useSignaling.ts), WebRTC peer connections
(useWebRTC.ts), and media streams (useUserMedia.ts). This keeps your React
components clean and focused on rendering UI.
2. State Management: For managing the complex state of multiple peer
connections, their statuses, and media streams, a state management library (like
Zustand, Jotai, or Redux Toolkit) can be very helpful. Alternatively, React Context
combined with useReducer can manage state effectively without external
dependencies.
3. Peer Connection Map: When implementing the multi-peer mesh, use a Map or a
plain object on the client to store RTCPeerConnection instances, keyed by the peer's
socketId. This makes it easy to add, access, and remove connections as peers join
or leave.
WebRTC Best Practices
1. Resource Cleanup: WebRTC objects and media streams are not automatically
garbage-collected when a component unmounts. Always implement cleanup
logic. When a peer disconnects or the user hangs up, explicitly call
peerConnection.close(). For media streams, iterate through stream.getTracks() and
call track.stop() on each one to release the camera and microphone.
2. ICE Trickling: Instead of waiting for all ICE candidates to be gathered before
sending the offer/answer, send them to the other peer as they are discovered. This
technique, known as "trickle ICE," can significantly speed up connection
establishment.

Security and Scalability
1. Signaling Security: In a production environment, your WebSocket server should be
secured. While not required for this task, consider how you might add
authentication to your signaling server to prevent unauthorized users from joining
rooms.
2. Scalability Trade-offs: The mesh topology is simple but doesn't scale well. A 4-
person call requires each client to manage 3 upstream and 3 downstream
connections. For larger calls, services typically use a Selective Forwarding Unit
(SFU) or a Multi-point Control Unit (MCU) to route media through a central server,
reducing client-side load. You should be prepared to discuss these alternatives.
FAQ
1. Q: Do I need a TURN server?
1. A: For this task, a STUN server (as specified in env-example) is sufficient, as
most testing will occur on local networks or environments where NAT
traversal is straightforward. A TURN server is only needed for complex
corporate or symmetric NAT scenarios, which are out of scope. (Related to
env-example)
2. Q: Can I use a library like simple-peer?
1. A: Yes, you can use a WebRTC abstraction library. However, be prepared to
explain the underlying WebRTC APIs and concepts it abstracts away (e.g.,
RTCPeerConnection, addTrack, offer/answer flow).
3. Q: How do I test a multi-peer call by myself?
1. A: You can open the same room URL in multiple browser tabs or windows.
Each tab will act as a separate peer. Note that your browser may limit
camera/microphone access to one tab at a time, but you can still test the
connection logic.
4. Q: My video connection isn't establishing. What should I check?
1. A: First, check your browser's developer console for any WebRTC or
WebSocket errors. Second, verify that signaling messages (offer, answer,
ice-candidate) are being correctly sent and received via the WebSocket
server. Third, check the WebRTC internals page in your browser
(chrome://webrtc-internals for Chrome) for detailed connection status and
diagnostics.
5. Q: How do I manage state when a new peer joins a room that already has active
participants?

1. A: When a new peer joins, the server should ideally send them a list of existing
participant IDs in that room. The new peer then initiates a P2P connection
with each of them. Similarly, the server should notify all existing peers of the
new arrival, so they can initiate a connection back. This ensures the mesh is
fully connected. (Related to multi-peer-mesh)
6. Q: What are SDP (Session Description Protocol) offers and answers?
1. A: SDP is a format for describing multimedia session details. In WebRTC, an
'offer' is an SDP object from the calling peer describing their proposed media
capabilities (e.g., codecs, resolutions). The 'answer' is the receiving peer's SDP
response, indicating their own capabilities and agreeing on a common
format for the call.
7. Q: Why does the custom server server.ts need ts-node?
1. A: The Next.js dev command is optimized for compiling and running the
Next.js application code itself. Since server.ts is a custom entry point written
in TypeScript, we use ts-node to compile and execute it on the fly in the
development environment. For production builds, you would first compile
server.ts to JavaScript and then run it with Node.
